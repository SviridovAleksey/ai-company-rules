# Приложение C — Чеклист апрув-ревью документации

Выполняется **до старта разработки**, после завершения Фазы 1 и построения Roadmap.
Все три роли должны дать апрув. При наличии замечаний — документация возвращается на доработку к соответствующему специалисту.

---

## Architect

- [ ] Архитектурный подход соответствует функциональным и нефункциональным требованиям
- [ ] Технологический стек обоснован и задокументирован в ADR
- [ ] Все значимые trade-offs зафиксированы в `docs/architecture/adr/`
- [ ] Компоненты системы и их взаимодействие описаны в `docs/architecture/overview.md`
- [ ] Архитектурные риски выявлены и имеют план митигации
- [ ] Документация достаточна для начала разработки без архитектурных вопросов

---

## Developer

- [ ] Все требования реализуемы в рамках выбранного стека
- [ ] Нет противоречий между `docs/business/`, `docs/sa/`, `docs/architecture/`, `docs/qa/`, `docs/devops/`
- [ ] Нет неопределённостей, блокирующих начало разработки
- [ ] `ROADMAP.md` покрывает все требования и разбит на реализуемые этапы
- [ ] Критерии готовности каждого этапа Roadmap чётко определены
- [ ] `docs/dev/technical-notes.md` зафиксировал все технические решения и открытые вопросы

---

## QA Lead

- [ ] Для каждого ключевого требования определены acceptance criteria в формате Given/When/Then
- [ ] Стратегия тестирования покрывает все уровни: unit, integration, e2e, performance, security
- [ ] Тестовые сценарии реализуемы и не зависят от неопределённых данных
- [ ] Нефункциональные требования имеют измеримые критерии приёмки
- [ ] `docs/qa/acceptance-criteria.md` готов к использованию как входные данные для TDD

- [ ] Кросс-валидация: нет противоречий между MD-файлами разных ролей
